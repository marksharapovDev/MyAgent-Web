## Два режима работы агента

### Автопилот (по умолчанию для больших задач)

Агент получает большой этап (например "дизайн всех страниц") и работает
самостоятельно, обращаясь к тебе только на ключевых точках.

**Как работает:**
- Берёт этап из ProjectPlan
- Разбивает на подзадачи сам
- Выполняет подзадачи последовательно
- Чекин: присылает результат пачки (3-5 подзадач) на ревью
- После одобрения — следующая пачка
- Интервал чекинов: каждые 30-60 минут реальной работы

**Когда автопилот сам останавливается и спрашивает:**
- Закончил пачку подзадач → ревью
- Встретил неясность, которую не может решить из контекста
- Нашёл конфликт с предыдущими решениями
- Требуется необратимое действие (деплой на прод, мерж в main)
- Задача оказалась сложнее, чем ожидалось → предлагает пересмотреть план

**Пример в Telegram:**
```
Бот: "📋 Автопилот: этап «Дизайн» (4 страницы)

      Начинаю с главной страницы. На основе неё
      сделаю остальные. Вернусь через ~30 мин
      с результатами.

      [⏸ Пауза] [🔍 Детальный режим]"

### Детальный режим

Ты работаешь с агентом в реальном времени, каждый шаг —
обсуждение и одобрение. Для точечных правок, сложных
дизайн-решений, или когда хочешь полный контроль.

**Как работает:**
- Одна подзадача за раз
- Показывает результат сразу после выполнения
- Ждёт явное одобрение/правки перед следующей
- Задаёт вопросы при любой неясности

**Переключение между режимами — в любой момент:**
- `/autopilot` — "дальше работай сам, вернись с результатами"
- `/detail` — "стоп, хочу контролировать каждый шаг"
- По кнопке в интерфейсе

```
Ты:  /detail

Бот: "Переключился в детальный режим.
      Текущий этап: Дизайн → страница «Каталог»

      Сейчас работаю над сеткой товаров.
      Показать текущий вариант?"
```

## Жизненный цикл проекта

### Фаза 0: Создание проекта

Ты создаёшь проект через /new. Загружаешь ТЗ от клиента
(текстом, голосом, PDF-файлом). Агент сохраняет в context_docs.

### Фаза 1: Планирование → ProjectPlan

Это НОВЫЙ модуль — `ProjectPlanner`. Самая важная фаза.

```typescript
// packages/core/src/planner/project-planner.ts

interface ProjectPlan {
  id: string;
  project_id: string;
  version: number;

  // Общее описание проекта
  summary: string;

  // Этапы (phases), каждый содержит задачи
  phases: Phase[];

  // Текущее состояние
  current_phase: number;
  current_task: number;
  mode: 'autopilot' | 'detail';

  created_at: Date;
  updated_at: Date;
}

interface Phase {
  id: string;
  title: string;           // "Дизайн", "Разработка", "Деплой"
  description: string;
  status: 'pending' | 'active' | 'done';
  tasks: PlanTask[];
}

interface PlanTask {
  id: string;
  title: string;           // "Дизайн главной страницы"
  description: string;
  type: 'design' | 'code' | 'deploy';
  depends_on: string[];    // ID задач, которые должны быть выполнены
  status: 'pending' | 'active' | 'done' | 'skipped';
  checkpoint: boolean;     // Нужно ли ревью после этой задачи
  variants: number;        // Сколько вариантов генерировать (для дизайна)
  estimated_minutes: number;
}
```

### Фаза 2-4: Выполнение через AutoPilot

```typescript
// packages/core/src/planner/autopilot.ts

class AutoPilot {
  private plan: ProjectPlan;
  private mode: 'autopilot' | 'detail';

  async run(): Promise<void> {
    for (const phase of this.plan.phases) {
      if (phase.status === 'done') continue;

      phase.status = 'active';
      await this.savePlan();

      for (const task of phase.tasks) {
        if (task.status === 'done') continue;

        // Проверяем зависимости
        const depsReady = task.depends_on.every(
          depId => this.findTask(depId).status === 'done'
        );
        if (!depsReady) {
          task.status = 'skipped';
          continue;
        }

        // Выполняем задачу
        task.status = 'active';
        await this.savePlan();

        const result = await this.executeTask(task);

        // Чекпоинт — спрашиваем пользователя
        if (task.checkpoint || this.mode === 'detail') {
          await this.requestReview(task, result);
          // Ждём ответа (может быть минуты или часы)
          const feedback = await this.waitForFeedback(task.id);

          if (feedback.action === 'approve') {
            task.status = 'done';
          } else if (feedback.action === 'revise') {
            // Возвращаемся к задаче с правками
            await this.reviseTask(task, feedback.text);
            continue; // Повторяем цикл для этой задачи
          } else if (feedback.action === 'redo') {
            task.status = 'pending';
            continue;
          }
        } else {
          // Не чекпоинт в автопилоте — просто идём дальше
          task.status = 'done';
        }

        await this.savePlan();
      }

      phase.status = 'done';
      await this.savePlan();

      // Уведомляем о завершении этапа
      await this.notifyPhaseComplete(phase);
    }
  }

  async executeTask(task: PlanTask): Promise<TaskResult> {
    switch (task.type) {
      case 'design':
        // Для дизайна: генерируем N вариантов если указано
        if (task.variants > 1) {
          const variants = [];
          for (let i = 0; i < task.variants; i++) {
            variants.push(await designPipeline.run(task, i));
          }
          return { type: 'design_variants', variants };
        }
        return designPipeline.run(task);

      case 'code':
        return codePipeline.run(task);

      case 'deploy':
        return deployPipeline.run(task);
    }
  }

  // Переключение режима
  async setMode(newMode: 'autopilot' | 'detail'): Promise<void> {
    this.mode = newMode;
    this.plan.mode = newMode;
    await this.savePlan();

    if (newMode === 'detail') {
      // Останавливаемся после текущей подзадачи
      // и показываем промежуточный результат
    }
  }
}
```

### Фаза 5: Поддержка

После деплоя проект переходит в режим поддержки.
Ты можешь в любой момент написать агенту:
- "Клиент просит поменять телефон на сайте"
- "Добавь новый товар в каталог"
- "На мобилке баг — кнопка не нажимается"

Каждое такое обращение — мини-цикл:
discussion → execute → review → deploy.

## Гибкость плана

**Ключевой принцип: план — это рекомендация, не жёсткая структура.**

```
Ты:  (в середине этапа дизайна)
      "Стоп, давай сначала сделаем минимальную версию
      только с главной и контактами, а остальное потом"

Бот: "Понял, обновляю план:

      Этап 1: Дизайн (сокращён)
        1.1 Главная (3 варианта) ← уже сделано ✅
        1.2 Контакты ← следующая
        ----
        1.3 О компании → отложено
        1.4 Услуги → отложено
        1.5 Галерея → отложено
        1.6 Отзывы → отложено

      Этап 2: Разработка (только главная + контакты)
      Этап 3: Деплой

      [✅ Ок] [✏️ Другие правки к плану]"
```

Также можно в любой момент:
- Добавить этап или задачу
- Удалить или отложить задачу
- Изменить порядок
- Разбить задачу на более мелкие
- Объединить мелкие в одну

```sql
CREATE TABLE project_plans (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id  UUID REFERENCES projects(id) ON DELETE CASCADE,
  version     INT NOT NULL DEFAULT 1,
  plan        JSONB NOT NULL,              -- Полная структура ProjectPlan
  status      TEXT DEFAULT 'active',       -- active | completed | archived
  mode        TEXT DEFAULT 'autopilot',    -- autopilot | detail
  created_at  TIMESTAMPTZ DEFAULT now(),
  updated_at  TIMESTAMPTZ DEFAULT now()
);

## Масштабирование: дорожная карта

### v1.0 — MVP (месяц 1)
То, что мы проектируем сейчас:
- Один агент, один мозг (Claude)
- Telegram + базовый Desktop
- Design + Code + Deploy pipelines
- ProjectPlan + AutoPilot
- Голосовой ввод

### v1.5 — Улучшения (месяц 2-3)
- Мульти-вариантность дизайна (2-3 варианта)
- Компонентная библиотека (агент переиспользует компоненты между проектами)
- Умный контекст: pgvector поиск по документации
- Desktop: полноценный UI с редактором документации
- Уведомления о деплоях и ошибках

### v2.0 — Мульти-агентность (месяц 4-6)
Вместо одного агента, который делает всё — специализированные агенты:

```
Orchestrator (главный)
  ├── DesignAgent      — специализируется на UI/UX
  ├── CodeAgent        — специализируется на Next.js/React
  ├── DeployAgent      — специализируется на инфраструктуре
  └── QAAgent          — проверяет работу других агентов
```

**Как это работает:**
- Orchestrator получает план и распределяет задачи
- DesignAgent делает дизайн, CodeAgent ждёт
- Когда дизайн одобрен, CodeAgent начинает работу
- QAAgent проверяет результаты каждого агента
- Если QA нашёл проблему — возвращает задачу обратно

**Технически:**
- Каждый агент — отдельный системный промпт с узкой специализацией
- Общение между агентами через очередь задач в Supabase
- Orchestrator решает кого задействовать
- Все агенты используют один Claude API, но с разными промптами

### v3.0 — Другие сферы жизни (месяц 6+)

Архитектура агента изначально модульная — pipelines можно добавлять:

```
packages/core/src/pipelines/
  ├── design.pipeline.ts        # v1.0
  ├── code.pipeline.ts          # v1.0
  ├── deploy.pipeline.ts        # v1.0
  ├── finance.pipeline.ts       # v3.0 — учёт расходов, инвойсы
  ├── health.pipeline.ts        # v3.0 — трекинг, напоминания
  ├── calendar.pipeline.ts      # v3.0 — планирование дня
  └── learning.pipeline.ts      # v3.0 — план обучения
```

**Финансы:**
- Интеграция с банковским API (Tinkoff, Sber)
- Трекинг доходов от проектов
- Автоматические инвойсы клиентам
- "Сколько я заработал в этом месяце?" → ответ с графиком

**Здоровье/спорт:**
- Интеграция с Apple Health / Google Fit
- Напоминания о тренировках
- Трекинг привычек
- "Напомни выпить воду каждые 2 часа"

**Календарь:**
- Интеграция с Google Calendar
- Планирование рабочего дня
- "Поставь созвон с клиентом на пятницу 15:00"
- Автоматическое распределение задач по дню

**Ключевой принцип масштабирования:**
Каждая новая сфера = новый pipeline + новые tools + новый промпт.
Ядро (TaskRouter, DiscussionSession, AutoPilot) остаётся тем же.
Telegram и Desktop автоматически поддерживают новые pipeline
через TaskRouter.
